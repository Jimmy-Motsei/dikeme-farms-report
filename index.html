Project Overview

The incident originated from outdated WooCommerce template overrides in the production theme. Immediate efforts were directed towards stabilizing the storefront, safeguarding customer data, and ensuring a reliable rollback path while maintenance mode was active.

Job Timeline

  • 12h45 — Site placed in maintenance mode to protect live visitors.
  • 13h05 — Initiated investigations via Afrihost File Manager and enabled debugging through wp-config to collect error details.
  • 13h30 — Full backup created and redundant archives cleared.
  • 14h10 — Attempted WordPress core update (v6.5.6 → v6.8.3); failure traced to obsolete theme templates.
  • 14h45 — Replacement templates deployed through a child theme.
  • 14h58 — Fatal error resolved; maintenance mode lifted after QA pass.

Troubleshooting & Stabilization

  • Placed the website in Maintenance Mode and notified stakeholders.
  • Accessed File Manager (Afrihost) to inspect logs and server files.
  • Updated wp-config.php to enable test/debug mode for verbose stack traces.
  • Performed a full site backup for safe rollback and disaster recovery.
  • Deleted legacy backups to maintain disk usage thresholds within hosting limits.
  • Attempted WordPress update (v6.5.6 → v6.8.3) to bring core in line with security fixes.
  • Identified outdated WooCommerce template overrides as the breaking change driver.
  • Compressed and downloaded public_html to local storage for offline analysis.

Child Theme & Template Refresh

  • Created a fresh child theme to isolate customizations from vendor code.
  • Added baseline style.css and functions.php with enqueue hooks.
  • Activated the child theme in WordPress Admin to assume control of overrides.
  • Rebuilt WooCommerce templates to align with the latest plugin version.
  • Cleared caches and revalidated cart, checkout, and account flows end-to-end.
  • Issue fully resolved at 14:58
